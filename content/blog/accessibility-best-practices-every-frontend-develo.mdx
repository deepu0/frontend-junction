Accessibility isn't a feature—it's a fundamental aspect of good web development. With WCAG 2.2 now widely adopted and stricter legal requirements in place, building accessible interfaces is both an ethical responsibility and a business necessity. Here's what every frontend developer needs to know in 2026.

## Semantic HTML: The Foundation

The single most impactful thing you can do for accessibility is write semantic HTML. Screen readers rely on structure to navigate pages, and semantic elements communicate meaning that visual styling alone cannot convey.

Instead of `<div>` for everything, reach for proper elements:

```jsx
// Instead of this
<div className="btn" onClick={handleSubmit}>Submit</div>

// Do this
<button type="submit">Submit</button>
```

Buttons get native keyboard support, focus states, and screen reader announcements for free. Use `<nav>` for navigation regions, `<main>` for primary content, `<article>` for self-contained content, and `<aside>` for supplementary material. This structure lets users jump between regions quickly.

## Keyboard Navigation and Focus Management

Every interactive element must be reachable and operable via keyboard alone. The golden rule: if you can click it, you must be able to tab to it.

Modern React applications often break focus when components unmount. Use proper focus management:

```jsx
function Modal({ isOpen, onClose, children }) {
  const modalRef = useRef(null);

  useEffect(() => {
    if (isOpen) {
      modalRef.current?.focus();
      document.body.style.overflow = 'hidden';
    }
    return () => {
      document.body.style.overflow = '';
    };
  }, [isOpen]);

  if (!isOpen) return null;
  
  return (
    <div 
      role="dialog" 
      aria-modal="true"
      ref={modalRef}
      tabIndex={-1}
      onKeyDown={(e) => e.key === 'Escape' && onClose()}
    >
      {children}
    </div>
  );
}
```

This pattern traps focus within the modal when open and returns it to the trigger when closed—essential for both screen reader and keyboard users.

## ARIA: Use It Sparingly

ARIA (Accessible Rich Internet Applications) can fix accessibility issues, but it can't create semantic meaning where none exists. The rule: if you can use native HTML, do so. Only reach for ARIA when you must.

Three crucial ARIA rules to remember:

1. **Never change semantics** — Don't add `role="button"` to a `<div>`; use `<button>` instead.
2. **Never remove semantics** — Don't add `role="presentation"` to semantic elements unless you're intentionally removing their meaning.
3. **Always maintain accessibility** — If you hide something from screen readers with `aria-hidden`, ensure keyboard users aren't also locked out.

```jsx
// Good: Adding meaning where HTML falls short
<span aria-label="3 notifications" className="badge">3</span>

// Bad: Trying to fix missing semantics
<div role="button" onClick={handleClick}>Click me</div>
```

## Color and Visual Design

Color blindness affects approximately 8% of men and 0.5% of women. Never rely solely on color to communicate information. Add secondary indicators like icons, patterns, or text labels:

```css
/* Don't do this */
.status-success { color: green; }

/* Do this instead */
.status-success { 
  color: green; 
  background-image: url('/check-icon.svg');
  background-repeat: no-repeat;
}

/* Also ensure 4.5:1 contrast ratio for text */
/* WCAG AA requires: 
   - 4.5:1 for normal text
   - 3:1 for large text (18px+ or 14px+ bold)
*/
```

Tools like the WCAG Color Contrast Checker and browser dev tools now include built-in accessibility auditing—use them during development.

## Testing: Automate and Manual Check

Automated tools catch only about 30% of accessibility issues. Run audits in Chrome's Lighthouse or Axe DevTools, but also test manually:

- **Tab through** your entire interface with no mouse
- **Disable styles** and navigate by headings
- **Use a screen reader** (VoiceOver, NVDA) to experience your site as users do
- **Zoom to 200%** and verify nothing breaks

In 2026, accessibility testing should be part of your CI pipeline. Tools like `axe-core` and `lint-staged` can catch issues before they reach production.

## Conclusion

Accessibility is a continuous practice, not a checkbox. Start with semantic HTML, ensure keyboard operability, use ARIA only when necessary, design with color accessibility in mind, and test with real tools. The effort you invest makes the web usable for everyone—and that's worth building for.
