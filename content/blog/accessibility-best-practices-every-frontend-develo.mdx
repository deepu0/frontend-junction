Accessibility isn't a nice-to-have—it's essential. With over 1 billion people having some form of disability, building accessible interfaces isn't just ethical; it's good business. Yet many frontend developers still treat accessibility as an afterthought. Let's change that.

## Semantic HTML: The Foundation

The simplest way to make your site accessible starts with proper HTML. Semantic elements like `<button>`, `<nav>`, `<main>`, and `<article>` give screen readers the context they need. Don't use `<div>` for everything.

```jsx
// ❌ Bad: No semantic meaning
<div onClick={handleClick} className="btn">Click me</div>

// ✅ Good: Built-in accessibility
<button onClick={handleClick}>Click me</button>
```

Buttons get keyboard support, focus states, and ARIA roles for free. Every time you reach for a `<div>` as a click target, ask yourself: should this be a semantic element instead?

## ARIA: Use It Wisely

ARIA (Accessible Rich Internet Applications) can bridge gaps when semantic HTML isn't enough. But use it sparingly—ARIA should supplement semantic HTML, not replace it.

The first rule of ARIA: don't use it if a native element exists. The second rule: if you do use it, use it correctly.

```jsx
// ❌ Bad: Redundant ARIA
<button role="button" aria-label="Submit form">Submit</button>

// ✅ Good: Only when needed
<div 
  role="combobox" 
  aria-expanded={isOpen} 
  aria-controls="dropdown-list"
>
  Select an option
</div>
```

## Keyboard Navigation Matters

Users navigates websites without a mouse. Ensure all interactive elements are reachable and operable via keyboard. This means:

- Logical tab order (no skipping elements)
- Visible focus indicators (never `outline: none` without replacement)
- Skip links for navigation menus
- Keyboard陷阱 prevention

```jsx
// ✅ Good: Focusable with visible state
const Button = ({ children, onClick }) => (
  <button
    onClick={onClick}
    className="focus-visible:ring-2 focus-visible:ring-blue-500"
  >
    {children}
  </button>
);

// Skip link example
<a href="#main-content" className="sr-only focus:not-sr-only">
  Skip to main content
</a>
```

Modern CSS frameworks like Tailwind provide `focus-visible` utilities to show focus states only when navigating via keyboard, keeping the visual design clean for mouse users.

## Color and Contrast

WCAG 2.1 requires a minimum contrast ratio of 4.5:1 for normal text and 3:1 for large text. This isn't optional—it's the law in many jurisdictions.

```css
/* ✅ Good: Meets WCAG AA */
.text-primary {
  color: #1a1a2e; /* ratio: 14.74:1 on white */
}

/* ❌ Bad: Hard to read */
.text-muted {
  color: #6b7280; /* ratio: 3.02:1 - barely passes */
}
```

Tools like the WebAIM Contrast Checker help verify your colors. Also, never rely on color alone to convey information—pair colors with icons or text labels.

## Testing: Beyond Automated Tools

Automated accessibility testing catches only about 30-50% of issues. Manual testing is essential:

- Navigate with only a keyboard
- Disable images and test alt text
- Use a screen reader (VoiceOver, NVDA)
- Zoom to 200% and verify layout doesn't break
- Test with motion reduced (prefers-reduced-motion)

```css
/* Respect user preferences */
@media (prefers-reduced-motion: reduce) {
  * {
    animation: none !important;
    transition: none !important;
  }
}
```

## Conclusion

Accessibility is an ongoing process, not a checkbox. Start with semantic HTML, add proper ARIA when needed, ensure keyboard navigation works, verify color contrast, and test with real users. Small investments in accessibility pay dividends in usability for everyone—and sometimes, it's the law.

The best accessible code is code that doesn't require extra workarounds. Build it in from the start.
