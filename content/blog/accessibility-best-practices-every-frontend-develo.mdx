Accessibility isn't a nice-to-have—it's essential. Yet many developers treat it as an afterthought, often retrofitting it late in development when it costs significantly more to implement. The good news? Building accessible interfaces from the start doesn't add much overhead, and the techniques benefit everyone, not just users with disabilities.

## Semantic HTML: Your Foundation

The easiest way to improve accessibility is using semantic HTML correctly. Screen readers rely on HTML semantics to navigate and describe content. Using `<button>` for actions and `<a>` for links isn't just cleaner—it enables keyboard navigation and proper screen reader announcements out of the box.

```jsx
// ❌ Don't use divs for interactive elements
<div onClick={handleSubmit}>Submit</div>

// ✅ Use semantic elements
<button onClick={handleSubmit} type="submit">Submit</button>
```

Beyond accessibility, semantic HTML improves SEO and gives you free functionality like form validation and focus management.

## Focus Management: The Keyboard Connection

Keyboard users navigate your interface by moving focus between interactive elements. Proper focus management ensures they can access everything. The golden rule: visible focus indicators should never be removed without a replacement.

```css
/* Always maintain visible focus */
:focus-visible {
  outline: 2px solid #2563eb;
  outline-offset: 2px;
}

/* Remove default outline only when providing custom focus styles */
.custom-button:focus-visible {
  outline: none;
  box-shadow: 0 0 0 3px rgba(37, 99, 235, 0.5);
}
```

When building single-page applications or modals, manage focus programmatically to keep keyboard navigation within the active context:

```tsx
function Modal({ isOpen, onClose, children }) {
  const modalRef = useRef(null);

  useEffect(() => {
    if (isOpen) {
      // Focus the modal when opened
      modalRef.current?.focus();
      // Trap focus within modal
    }
  }, [isOpen]);

  if (!isOpen) return null;
  return (
    <div 
      role="dialog" 
      aria-modal="true"
      ref={modalRef}
      tabIndex={-1}
    >
      {children}
    </div>
  );
}
```

## ARIA: Use It Correctly

ARIA (Accessible Rich Internet Applications) fills gaps when HTML falls short—but only when used correctly. The first rule of ARIA: don't use it if semantic HTML works. The second rule: if you use ARIA, it must conform to the specification.

```jsx
// ❌ Wrong: redundant or incorrect ARIA
<button aria-label="Click here"></button>
<div role="button" onClick={...}></div>

// ✅ Correct: clear labels for icon-only buttons
<button aria-label="Close menu">
  <CloseIcon />
</button>
```

Common patterns like expandable sections need `aria-expanded`, progress bars need `aria-valuenow`, and live regions need appropriate roles. Always test with a screen reader to verify your ARIA implementations work.

## Color and Contrast: Beyond Aesthetics

WCAG 2.1 requires a minimum 4.5:1 contrast ratio for normal text and 3:1 for large text. This isn't just about compliance—it ensures readability in various lighting conditions and for users with visual impairments.

Use tools like the Chrome DevTools accessibility pane or WebAIM's contrast checker to verify. Beyond contrast, avoid conveying information through color alone—add icons or text labels:

```jsx
// ❌ Color-only status indicator
<div className="status error"></div>

// ✅ Color with text and icon
<div className="status error" role="status">
  <ErrorIcon aria-hidden="true" />
  <span>Payment failed</span>
</div>
```

## Testing: Finding Issues Early

Automated tools catch roughly 30-40% of accessibility issues. For the rest, combine multiple approaches:

1. ** axe DevTools ** or ** Lighthouse ** for automated scanning
2. ** Keyboard-only navigation ** to experience the site as keyboard users do
3. ** Screen reader testing ** with VoiceOver (macOS) or NVDA (Windows)
4. ** Zoom testing ** at 200% to ensure layouts don't break

Integrate accessibility testing into your workflow—eslint-plugin-jsx-a11y catches many issues during development, andaxe-core can run in your CI pipeline.

## Conclusion

Accessibility benefits everyone: better SEO, improved usability on mobile, and code that's easier to maintain. Start with semantic HTML, manage focus properly, use ARIA correctly, verify color contrast, and test with real users. These practices take minimal time upfront compared to retrofitting later—and they make your applications genuinely usable for the full diversity of your users.
